0. 프로젝트 정체성

Hyper-AI는
"복잡한 AI 작업을 구조적으로 조합할 수 있는 실행 엔진"
이며,
장기적으로는
"구조 언어(Definition Language) + 실행 플랫폼(Runtime Platform)"
이다.

설계도는 목적이 아니라,
엔진을 사용하는 방식(입력물)이다.

----------------------------------------------------------------------
1. 핵심 철학

1) 조합 중심
   - Hyper-AI의 본질은 AI 실행이 아니라
     AI 작업의 구조적 조합 능력이다.

2) 안정성 우선
   - 안전 > 재현성 > 관측성 > 확장성

3) 계약 중심
   - 모든 Node는 명시적 계약을 가진다.
   - 계약 없는 실행은 허용하지 않는다.

4) 설계 단계 검증
   - 실행 전에 구조를 검증한다.
   - "돌려봐야 아는 구조"를 허용하지 않는다.

----------------------------------------------------------------------
2. Hyper-AI Definition Language v0.1 (최소 개념 문법)

이 언어는 AI 작업 구조를 표현하기 위한 설계 언어이다.
실행은 Runtime이 담당한다.

----------------------------------------------------------------------
2.1 Node (최소 실행 단위)

각 Node는 반드시 가진다:

- id (내부 고유 식별자)
- name (사람이 읽는 이름)
- kind (gate / plugin / ai)
- ref (실제 구현 참조)
- contract
    - in (입력 스키마)
    - out (출력 스키마)
    - fail_codes (실패 코드 목록)

Node는 계약이 있는 실행 단위이다.

----------------------------------------------------------------------
2.2 Connection (연결)

- 출력 → 입력 연결은 명시적으로 정의한다.
- 자동 추측 연결 금지.
- 타입 불일치 시 설계 단계에서 차단.
- 필드 매핑은 명시적으로 작성한다.

----------------------------------------------------------------------
2.3 Flow (흐름 구조)

최소 지원 구조:

1) Sequential
   A → B → C

2) Conditional
   조건 기반 분기
   IF → A
   ELSE → B

3) Parallel
   여러 노드 동시 실행 후 병합

4) Error Routing
   실패 시:
   - stop
   - retry
   - route to node

----------------------------------------------------------------------
3. Definition 파일 최소 구성 요소

각 Definition은 반드시 포함해야 한다:

- format
- version
- name
- settings
- nodes
- edges
- flow

----------------------------------------------------------------------
4. JSON 정본 스키마 예시 (v0.1)

{
  "format": "hyperai.definition",
  "version": "0.1.0",
  "name": "example-definition",
  "settings": {
    "ai": {
      "provider": "openai",
      "model": "gpt-5",
      "temperature": 0.2,
      "timeout_ms": 20000
    },
    "policy": {
      "sandbox_plugins": true,
      "max_retries": 1
    }
  },
  "nodes": [
    {
      "id": "n1",
      "name": "collect_input",
      "kind": "gate",
      "ref": { "gate_key": "inputs.collect" },
      "contract": {
        "in": { "type": "object" },
        "out": { "type": "object" },
        "fail_codes": ["INVALID_INPUT"]
      }
    }
  ],
  "edges": [],
  "flow": {
    "start": "n1",
    "on_error": {
      "default": { "action": "stop" }
    },
    "branches": [],
    "parallel": []
  }
}

----------------------------------------------------------------------
5. 설계 단계 검증 규칙 (v0.1)

Definition 저장 시 반드시 검사:

1) 모든 Node에 contract 존재
2) 모든 연결 타입 호환
3) start 노드 존재
4) 순환 구조 유효성 검사
5) 실패 코드 누락 여부 검사
6) 버전 정보 존재 여부
7) 참조(ref) 등록 여부

하나라도 실패하면 실행 금지.

----------------------------------------------------------------------
6. UI 전략

기본 방향: B (시각적 조합 우선)

그러나 반드시:

- JSON 정본 1개만 존재
- UI ↔ JSON 라운드트립 보장
- 텍스트 편집도 가능

정본은 항상 JSON.

----------------------------------------------------------------------
7. 장기 구조 비전

Hyper-AI는:

- 계약 기반 구조 언어
- 설계 단계 정적 검사
- 실행 런타임
- 정의 파일 버전 관리
- 안전한 플러그인 확장

으로 구성되는
AI 구조 플랫폼이다.
