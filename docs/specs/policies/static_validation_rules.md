======================================================================
Hyper-AI Static Validation Rules v0.2
(설계 단계 정적 검사 규칙 – 복붙용 단일 블록)
======================================================================

목적:
실행 전에 구조적 오류를 100% 차단한다.
"돌려봐야 아는 구조"를 허용하지 않는다.

----------------------------------------------------------------------
1. 구조 무결성 검사 (Structural Integrity)

1.1 format 유효성
- format 필드 존재
- version 존재
- name 존재

1.2 Node 무결성
- 모든 node는 id 존재
- id는 유일해야 함 (중복 금지)
- name 존재
- kind는 gate / plugin / ai 중 하나
- ref 필수
- contract 필수

1.3 Start 노드 존재
- flow.start 존재
- start가 nodes에 실제 존재해야 함

----------------------------------------------------------------------
2. 계약 검사 (Contract Validation)

2.1 입력/출력 스키마 존재
- contract.in 존재
- contract.out 존재
- fail_codes 배열 존재

2.2 타입 명시
- in/out은 반드시 type 필드 포함
- object 타입이면 properties 명시
- required 필드가 properties 안에 존재해야 함

2.3 실패 코드 명확성
- fail_codes는 최소 1개 이상
- 중복 fail_code 금지
- 정의되지 않은 fail_code 사용 금지

----------------------------------------------------------------------
3. 연결 검사 (Edge Validation)

3.1 참조 유효성
- from.node_id 존재해야 함
- to.node_id 존재해야 함

3.2 타입 호환성
- from.out 구조와 to.in 구조 타입 호환 검사
- 필수 필드 누락 시 오류

3.3 필드 매핑 검사
- map 배열 존재
- from_field는 실제 출력 필드여야 함
- to_field는 실제 입력 필드여야 함
- 동일 입력 필드에 다중 매핑 금지

3.4 고립 노드 검사
- start로부터 도달 불가능한 node 존재 시 오류

----------------------------------------------------------------------
4. 흐름 검사 (Flow Validation)

4.1 순환 검사
- 무한 루프 구조 감지
- 명시적 루프가 아닌 순환은 금지

4.2 분기 완결성
- branches 조건은 명확한 node_id + field + op + value 포함
- then / else 모두 정의되어야 함
- 분기 대상 node 존재 확인

4.3 병렬 구조 검사
- parallel 블록 내 node_id 존재 확인
- 병합 지점 정의 필요
- 병합 없는 병렬 실행 금지

----------------------------------------------------------------------
5. 실패 처리 검사 (Error Handling Validation)

5.1 on_error 기본 전략 존재
- default action 정의 필수

5.2 override 검사
- override.node_id 실제 존재
- codes는 해당 node의 fail_codes 안에 있어야 함
- route 대상 node 존재 확인

5.3 재시도 제한 검사
- retry 시 max_retries 명시 필수
- max_retries는 0 이상 정수

----------------------------------------------------------------------
6. 전역 설정 검사 (Global Settings Validation)

6.1 AI 설정 존재
- provider 명시
- model 명시
- timeout_ms 양수

6.2 정책 설정
- sandbox_plugins boolean
- max_retries 정수

----------------------------------------------------------------------
7. 버전 관리 검사

7.1 definition version 존재
7.2 format version과 호환성 확인
7.3 미래 버전 감지 시 경고 또는 차단 정책 적용

----------------------------------------------------------------------
8. 금지 규칙 (Hard Stop Conditions)

다음 중 하나라도 발생하면 실행 금지:

- 계약 없는 노드
- 타입 불일치 연결
- 고립 노드 존재
- 순환 구조 오류
- 실패 코드 미정의 사용
- start 노드 없음
- ref 등록되지 않음

----------------------------------------------------------------------
9. 설계 원칙 요약

1) 암묵적 연결 금지
2) 추측 기반 타입 변환 금지
3) 계약 없는 실행 금지
4) 구조 검증 실패 시 실행 차단
5) 모든 오류는 명확한 reason_code로 표현
