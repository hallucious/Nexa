======================================================================
Hyper-AI Execution State Model v0.1
(실행 상태 모델 정의 – 복붙용 단일 블록)
======================================================================

목적:
Definition이 실행되는 동안
데이터 흐름, 상태 저장, 추적 방식을 명확히 정의한다.

정적 검사가 끝난 이후의 "실행 상태"를 다룬다.

----------------------------------------------------------------------
1. 기본 개념

Execution은 하나의 독립 실행 단위다.

각 실행은 다음을 가진다:

- execution_id (전체 실행 식별자)
- definition_version
- start_time
- end_time
- final_status (success / failed)

----------------------------------------------------------------------
2. 상태 계층 구조

Execution State는 3계층으로 구성된다:

2.1 Global State
- 실행 전체에 공유되는 읽기 전용 설정
- settings (model, policy 등)
- execution metadata

2.2 Node State
각 노드 실행마다 생성된다:

- node_id
- trace_id
- input_snapshot
- output_snapshot
- status (pending / running / success / failed)
- reason_code (실패 시)
- start_time
- end_time
- retry_count

2.3 Flow State
- 현재 실행 경로
- 분기 선택 기록
- 병렬 대기 목록
- 재시도 기록

----------------------------------------------------------------------
3. 데이터 흐름 모델

3.1 입력 전달

- Node는 이전 Node의 output_snapshot을 입력으로 받는다.
- 전달 시 매핑 규칙(map) 적용.
- 입력은 Node State에 snapshot으로 저장된다.
- 원본 데이터는 수정하지 않는다 (불변성 유지).

3.2 출력 생성

- Node 실행 결과는 output_snapshot에 저장된다.
- 다음 노드로 전달될 때 복사본 사용.
- 이전 노드의 상태는 변경되지 않는다.

3.3 불변성 원칙

- 실행 중 과거 Node State 수정 금지.
- 모든 상태는 append-only 기록.
- 구조는 실행 중 변경 불가.

----------------------------------------------------------------------
4. 분기 상태 관리

4.1 Conditional

- 조건 평가 결과 기록.
- 선택된 경로 저장.
- 선택되지 않은 경로는 실행되지 않음.

4.2 Parallel

- 병렬 노드 각각 독립 Node State 생성.
- 모든 병렬 노드 완료 후 병합 단계 진입.
- 병합 시 데이터 충돌 발생하면 실패 처리.

----------------------------------------------------------------------
5. 실패 상태 처리

5.1 실패 기록

- 실패 Node State는 반드시 저장.
- reason_code 명시.
- message 포함.

5.2 Retry 처리

- retry_count 증가.
- retry도 별도 trace_id 생성.
- 이전 실패 기록 보존.

5.3 Route 처리

- 실패 시 Flow State에 경로 변경 기록.
- 자동 복구 금지 (정의된 정책만 허용).

----------------------------------------------------------------------
6. 실행 결과 구조

최종 Execution Result는 다음을 포함:

{
  "execution_id": "...",
  "definition_version": "...",
  "final_status": "success" | "failed",
  "node_results": [
      {
        "node_id": "...",
        "status": "...",
        "reason_code": "...",
        "trace_id": "..."
      }
  ],
  "path_trace": [...],
  "start_time": "...",
  "end_time": "..."
}

----------------------------------------------------------------------
7. 상태 저장 전략

v0.1 기준:

- 메모리 내 상태 유지
- 실행 종료 후 JSON 형태로 export 가능
- 향후 외부 저장소 확장 가능 (DB, 파일 시스템)

----------------------------------------------------------------------
8. 철학 요약

1) 상태는 불변이다.
2) 모든 실행은 추적 가능해야 한다.
3) 실패는 기록되고 숨기지 않는다.
4) 재시도는 새 실행으로 취급한다.
5) 구조와 상태는 분리된다.
