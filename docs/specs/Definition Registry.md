======================================================================
Hyper-AI Definition Registry Specification v0.1
(정의 저장·검색·버전관리·공유 체계 – 복붙용 단일 블록)
======================================================================

목적:
Definition(설계도)을 안전하게 저장, 검색, 버전 관리하고
재현 가능하게 실행할 수 있도록 한다.

Registry는 실행 엔진과 분리된 관리 계층이다.

----------------------------------------------------------------------
1. 기본 개념

Registry는 Definition의 저장소이다.
Definition은 불변 아티팩트(immutable artifact)로 취급한다.

Definition은 수정하지 않는다.
수정은 새로운 revision을 생성한다.

----------------------------------------------------------------------
2. Definition 식별 구조

각 Definition은 다음 4요소로 식별된다:

- namespace (조직/사용자 구분)
- name (설계도 이름)
- format_version (언어 버전)
- definition_revision (개별 수정 버전)

고유 키:

namespace/name:definition_revision

예:
acme/summarizer:1.2.0

----------------------------------------------------------------------
3. 저장 원칙

3.1 불변성
- 동일 revision 재업로드 금지
- 수정 시 반드시 revision 증가

3.2 무결성
- 업로드 시 정적 검사 통과 필수
- SHA256 해시 저장
- 변경 감지 가능해야 함

3.3 메타데이터 저장

각 Definition은 다음 메타 포함:

- created_at
- created_by
- description
- tags (검색용)
- dependency_list (사용된 subgraph / plugin / model)

----------------------------------------------------------------------
4. 검색 기능

Registry는 다음 기준으로 검색 가능:

- name
- namespace
- tag
- model 사용 여부
- plugin 사용 여부
- format_version
- 최근 수정일

----------------------------------------------------------------------
5. 버전 정책

5.1 최신 버전 조회
- 동일 namespace/name 중 가장 높은 revision 반환

5.2 특정 revision 조회
- 정확한 revision 명시 시 해당 버전만 반환

5.3 MAJOR 충돌
- format_version 불일치 시 경고 또는 차단

----------------------------------------------------------------------
6. 실행 연계

Runtime 실행 시:

1) Definition ID로 Registry 조회
2) 해당 revision 고정 로드
3) SHA 해시 검증
4) format_version 호환성 확인
5) 실행

----------------------------------------------------------------------
7. 의존성 관리

Definition은 사용한 Subgraph/Plugin/Model 버전을 명시적으로 고정한다.

예:

"dependencies": {
  "subgraphs": {
    "text/cleanup": "2.1.0"
  },
  "plugins": {
    "fact.checker": "1.4.2"
  }
}

의존성 버전은 자동 업데이트하지 않는다.
명시적 업그레이드만 허용한다.

----------------------------------------------------------------------
8. 공유 정책

8.1 공개/비공개 구분

- visibility: public | private | organization

8.2 공개 Definition은:
- 정적 검사 통과 필수
- 정책 엔진 검증 통과 필수

----------------------------------------------------------------------
9. 감사 로그 (Audit Log)

Registry는 다음을 기록한다:

- 업로드 기록
- 삭제 시도 기록
- 실행 호출 기록
- 권한 변경 기록

모든 기록은 수정 불가.

----------------------------------------------------------------------
10. 금지 규칙

- Definition 직접 수정 금지
- revision 덮어쓰기 금지
- 암묵적 의존성 업데이트 금지
- 미검증 Definition 실행 금지

----------------------------------------------------------------------
11. 철학 요약

1) Definition은 코드와 동등한 1급 아티팩트다.
2) 수정은 생성이다.
3) 실행은 항상 고정된 revision으로 이루어진다.
4) 공유는 검증 위에서만 허용된다.
5) 재현성은 Registry가 보장한다.
